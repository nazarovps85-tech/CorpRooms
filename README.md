## 🎯 Бизнес-контекст и проблематика
На текущий момент в компании отсутствует единый инструмент для управления переговорными комнатами, что приводит к конфликтам («накладкам» встреч), простаиванию помещений из-за забытых отмен и высоким временным затратам сотрудников (до 20 минут на организацию одной встречи).

**Главная цель проекта** — сократить время сотрудников на бронирование до **<2 минут** и повысить утилизацию (эффективность загрузки) переговорных комнат на **35%**.

## 🧠 Ключевые архитектурные и функциональные решения
В рамках проектирования были заложены следующие критически важные модули и требования:
* **Защита от Race Condition:** Блокировка одновременного бронирования одного и того же временного слота на уровне СУБД.
* **Автоотмена (No-Show):** Механизм автоматического освобождения комнаты, если организатор не подтвердил присутствие (чек-ин) на дверном планшете в течение 10 минут.
* **Двусторонняя синхронизация:** Интеграция с Exchange/Google Calendar для отображения внешних встреч в реальном времени.
* **Stateless-архитектура:** Возможность горизонтального масштабирования бэкенд-сервисов (подов в Kubernetes) под высокую конкурентную нагрузку.

## 📊 Структура спроектированной базы данных (ER)
Реляционная структура данных состоит из 4 ключевых сущностей, поддерживающих оптимизацию поисковых SQL-запросов на объёмах свыше 500 000 записей:
1. `Rooms` — физические помещения, их этажность, вместимость и массив оборудования.
2. `Users` — справочник сотрудников с ролевой моделью (`user` / `admin`).
3. `Bookings` — подтвержденные и аннулированные бронирования слотов.
4. `Booking_participants` — промежуточная сущность для реализации связи «многие со многими» (Many-to-Many) между встречами и участниками.

## 🛠 Спецификация REST API
В ТЗ детально спроектированы 10 основных эндпоинтов для взаимодействия компонентов системы:
* `POST /bookings` — Резервирование новой встречи организатором.
* `GET /rooms/search` — Интеллектуальный подбор комнат по фильтрам (дата, время, вместимость, оборудование).
* `PATCH /bookings/{booking_id}/status` — Отмена/аннулирование брони (с сохранением истории для аналитики).
* `POST & DELETE /bookings/{booking_id}/participants` — Управление составом участников встречи.
* `PATCH /rooms/{room_id}/status` — Перевод комнаты на обслуживание/ремонт (доступно только Администратору).
